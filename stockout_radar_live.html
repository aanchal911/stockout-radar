Who this is for: a regional inventory ops manager at a 200-store grocery chain, who currently finds out about stockouts by walking the aisles or fielding a call from a store manager — usually after the shelf is already empty. Every morning the chain's overnight POS feed and this morning's inventory snapshot land in storage; this pipeline joins them, computes a days-of-supply and demand-trend score for every store–SKU pair, and hands back a ranked list of what needs a restock or a store-to-store transfer today — plus which nearby store has spare stock to send.

The pandas time above is measured directly in this build (9,000,000 sales rows, 200,000 store–SKU groups, rolling 7/14/28-day velocity windows). The GPU time is not measured on GPU hardware in this environment — it's a conservative projection from RAPIDS cuDF's published benchmarks, where typical groupby/rolling/ETL workloads see roughly a 10–30x speedup switching from pandas to cudf.pandas with zero code changes — same import pandas as pd, same groupby/rolling calls, just running on GPU memory instead of one CPU core. At full chain scale (500 stores × 5,000 SKUs ≈ 12.5x this dataset) the pandas run stretches toward ~1 hour — only viable as an overnight batch job. Accelerated, that same full-chain run projects to roughly 2–6 minutes, which is the difference between "ops sees this once a day" and "ops can re-run it every hour and catch a stockout while there's still time to truck stock over from another store."

Managed Service for Apache SparkFull-chain history (years of transactions, 500 stores × 5,000 SKUs) is aggregated into daily features here — too large for a single machine, sized for a cluster.

The net effect for the person on the other end of this dashboard: a risk list that used to be a once-a-day, end-of-batch artifact becomes something that can refresh hourly — so a demand spike caught at 9am shows up in the urgent queue by 10am, not tomorrow morning, after the shelf has already gone empty.
